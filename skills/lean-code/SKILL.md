# 精简代码与模块边界

## 核心原则

先证明现有能力不能满足，再新增代码或抽象。让职责在第一次实现时就有边界，不要等文件失控后再治理；同时不要按行数机械拆散高度内聚的实现。

本 Skill 只管理结构复杂度，不替代安全、性能、正确性或业务规则审查。

**REQUIRED SUB-SKILL：**功能、修复、重构和行为变化进入执行阶段时，必须使用 `superpowers:test-driven-development`。新功能和修复先用失败测试锁定目标行为；纯重构先运行已有行为测试，覆盖不足时先补特征测试，再迁移。

## 工作模式与确认门槛

| 用户意图 | 模式 | 可否改代码 |
| --- | --- | --- |
| “看看哪里造轮子”“哪些能删” | 审计 | 否 |
| “给我拆分方案”“怎么重构” | 方案 | 否 |
| 已明确确认方案 | 执行 | 是 |
| 线上紧急修复且明确要求最小范围 | 最小变更 | 仅必要改动 |

“直接做”“今天上线”不能跳过方案确认。紧急修复也要先给出一页以内的最小方案；不要借机整理全文件或全仓。

## 实现前预检

新增功能、修复或扩展现有模块前，先回答：

1. 这是哪个领域或能力？本次新增了什么稳定变化原因？
2. 标准库、平台、框架、项目模块或现有依赖是否已经覆盖？
3. 代码应落在哪个现有模块？它通过什么接口被测试和调用？
4. 目标文件是否已经同时承担编排、领域规则、I/O、数据访问、第三方调用、缓存、权限、DTO/展示转换、UI 状态或框架装配中的多个职责？

### 文件增长停止线

- 预估单个手写生产文件超过 **400 行**：实现前写出一句单一职责说明和目标文件草图；无法用一个稳定职责解释时，先拆边界。
- 已有文件超过 **500 行**：不得继续加入新的稳定变化原因；本次只改原职责，或先给出最小切片迁移方案。
- 预估或实际超过 **800 行**：停止继续堆叠，先做职责审查；无法证明属于下述例外时，提交按职责拆分的方案并等待确认。

阈值是风险信号，不是拆分目标。以下内容可保持大文件：生成代码、声明式 schema、静态映射/数据、测试夹具、框架强制结构，以及接口很小且实现高度内聚的算法。必须说明共同变化原因；生成产物只改生成源，不手改产物。

## 复用与依赖门禁

按以下顺序查找能力，并在方案中保留检索证据：

1. 语言标准库。
2. 运行时、浏览器、数据库或云平台原生能力。
3. 当前框架的正式能力。
4. 项目内部模块和相邻实现。
5. 已安装依赖。
6. 仍在维护的成熟第三方库。
7. 只有前六项都不合适时才自定义实现。

日期/时区、加密与 ID、重试/退避、并发/队列、CSV/Excel/PDF/压缩包、Schema 解析、CLI 参数、URL 编码、缓存、状态机、金额精度、glob/diff/序列化等能力，不能凭记忆直接手写。至少记录一个真实候选，并比较：维护状态、类型支持、许可证、运行时兼容性、依赖成本，以及自维护成本。

稳定、清晰且只有 5～10 行的局部转换，不必为了“复用”增加依赖。若选择自定义实现，方案必须列出已查候选、缺失能力和需要自行承担的边界测试。

## 模块边界：追求深度，不追求文件数量

按领域职责命名模块，避免 `helpers`、`manager`、`processor`、`factory`、`common-utils` 等模糊容器。一个好模块应当：

- 只有一个稳定变化原因，相关知识保持局部；
- 用较小接口隐藏有实质复杂度的实现；
- 能通过公开接口测试，而不是暴露内部步骤；
- 删除后，复杂度会重新散落到多个调用者，而不是随模块一起消失。

执行**删除测试**：假设删除该抽象。若调用者只需内联一层转发且复杂度没有增加，就删除它；若规则会重复散落到多个真实调用者，该模块才有价值。

一个适配器通常不需要先造接口；两个真实适配器，或稳定外部契约/框架契约/可验证的测试与安全需要，才形成真实接缝。例外必须说明必要性、成本和为什么局部实现不足。

拆分前后各追踪一条典型调用链。若新增的跳转只转发参数、改名或重新导出，没有隐藏规则，就合并它。不要把 God File 变成一组浅层 wrapper。

## 审计与方案

审计时只读检查，按证据把结论归类：

| 分类 | 判定 | 动作 |
| --- | --- | --- |
| 删除 | 死代码、无意义 wrapper；静态调用、运行时注册、导出面、部署入口和兼容承诺均已核实 | 删除；证据不足则保持 |
| 复用 | 前六级已有能力覆盖 | 使用已有能力 |
| 延后抽象 | 只有一个调用者或变化点尚未发生 | 保持局部，记录抽取条件 |
| 按职责拆分 | 文件混合多个稳定变化原因 | 只迁移本次相关垂直切片 |
| 保持现状 | 边界清楚，拆分只增加跳转 | 说明共同变化原因 |

全仓减法审计可使用 `ponytail:ponytail-audit`；当前 diff 使用 `ponytail:ponytail-review`。输出沿用 `delete`、`stdlib`、`native`、`yagni`、`shrink` 分类。工具不可用时仍按本节完成。

针对 God File、跨模块领域接缝或整体架构治理，可选用 `improve-codebase-architecture`；普通局部功能不把它作为前置条件。未安装时可执行：

```bash
npx skills add https://github.com/mattpocock/skills --skill improve-codebase-architecture
```

方案必须包含：现状与调用证据、复用/依赖候选、本次目标与非目标、目标模块职责、兼容策略、测试与回滚。未获确认前不创建、移动或编辑业务代码。

## 执行：最小安全迁移

1. 新功能和修复先观察测试因目标行为缺失而失败；纯重构先运行并记录已有行为测试的 GREEN 基线，覆盖不足时先补特征测试。
2. 先复用现有能力；确需自定义时只实现候选比较后仍缺失的最小部分。
3. 一次迁移一个职责，保留必要兼容导出；不同时重命名、格式化或搬迁无关代码。
4. 每次迁移后运行相关测试、类型检查和调用方验证，再删除旧入口。
5. 对新模块做删除测试、单适配器检查和典型调用链检查。
6. 最后使用 `ponytail:ponytail-review`（若可用）复查 diff。

上线风险过高时，可以在原文件内完成经过测试的最小改动，但不得复制已有规则、重写平台能力或新增稳定变化原因；交付时说明为何本次不拆，以及下次治理触发条件。

## 交付结构证据

交付时报告：

- 最大文件的本次增量及最终行数；超过停止线时说明例外。
- 每个新增文件的唯一职责和真实调用者。
- 是否出现单调用者 wrapper、单适配器接口、模糊容器名或新的无价值跳转。
- 新增依赖或自定义基础能力的候选比较证据。
- 测试的 RED、GREEN、相关回归和类型检查结果。
- 生产代码、依赖和调用层级相对方案是增加还是减少。

## 红旗：停止并重新评估

- “先都写在一个文件，超过一千行再拆。”
- “这个轮子不难，自己写更快。”
- “拆成更多文件就算架构完成。”
- “先加接口/工厂，以后可能换实现。”
- “生成文件太长，也按行数拆。”
- “测试最后补，重构只是移动代码。”

出现任一项时，回到实现前预检；不要用赶工、未来复用或文件数量为决定辩护。
