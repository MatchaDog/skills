# 精简代码与模块边界

## 核心原则

先证明现有能力不能满足，再新增代码或抽象。复杂度本身是一种持续成本；没有第二个真实变化点的“通用性”通常只是猜测。

本 Skill 管理的是**结构复杂度**，不替代安全、性能、正确性或业务规则审查。普通的小功能开发不因文件行数自动进入本流程。

## 工作模式与确认门槛

| 用户意图 | 工作模式 | 可否改代码 |
| --- | --- | --- |
| “看看哪里造轮子”“审计一下”“哪些能删” | 审计 | 否 |
| “给我拆分方案”“怎么重构” | 方案 | 否 |
| 已明确确认方案，例如“确认，按此执行” | 执行 | 是 |
| 线上紧急修复且明确要求最小范围 | 最小变更 | 仅必要改动 |

不要把“直接做”“今天上线”理解为可以跳过方案确认。若用户的意图是紧急修复，给出一页以内的最小变更方案，并等待确认；不要借机做全文件或全仓整理。

## 审计：先找可复用能力，再谈重写或抽象

按以下顺序只读检查，并给出证据：

1. 语言标准库、运行时与平台原生能力能否满足；例如 Node、浏览器、数据库或框架已经提供的能力。
2. 项目已有依赖、公共模块、领域服务和相邻实现能否直接复用。
3. 当前需求是否存在至少两个**已经发生且变化方式不同**的调用场景。没有时，默认不创建接口、工厂、策略层、抽象基类或“通用”框架。框架契约、稳定的对外 API、明确可替换的基础设施边界，或可验证的测试/安全需求确有要求时可以例外，但方案必须说明必要性、成本和为什么局部实现不足。
4. 文件是否混合了编排、领域规则、I/O、数据访问、第三方调用、展示/DTO 转换等不同职责；不要只以行数判定。

如果可用，使用 `ponytail:ponytail-audit` 做全仓减法审计，或用 `ponytail:ponytail-review` 审查当前 diff。工具不可用时，仍按本 Skill 的分类完成审计。

### 审计输出格式

每项结论必须归入一类，并写出位置、证据、建议和风险：

| 分类 | 判定 | 动作 |
| --- | --- | --- |
| 删除 | 死代码、无意义 wrapper，且静态调用、运行时注册/配置、导出面、部署入口和兼容承诺均已核实无引用 | 删除；不以“以后可能用”保留；证据不足时保持现状 |
| 复用 | 标准库、平台、既有依赖或模块已覆盖 | 直接使用已有能力 |
| 延后抽象 | 仅一个调用者或变化点尚未出现 | 维持局部实现，记录抽取条件 |
| 按职责拆分 | 一个文件混杂多个稳定变化原因 | 为本次涉及职责建立最小目标模块 |
| 保持现状 | 当前边界清楚、重复不足以抵消迁移风险 | 说明为什么不动 |

## 方案：限定目标边界

在未获确认前，只提交方案，不创建、移动或编辑业务代码。方案必须包含：

1. **现状与证据：** 文件职责图、可复用能力、调用方和行为约束。
2. **本次目标：** 要删除、复用、延后或拆出的具体内容；明确不动哪些无关部分。
3. **目标边界：** 每个目标文件只描述一个职责和真实调用关系；`utils` 只能承载跨领域、无业务语义的纯工具，不能继续吸收订单、营销、权限等领域规则。
4. **兼容策略：** 导出、接口、错误语义、数据精度、顺序副作用与迁移期间的兼容入口。
5. **验证和回滚：** 需要新增或保留的行为测试、类型检查、相关集成测试，以及失败时如何回退。

不要提出“顺便整理整个目录”。当巨型文件中只有一个本次相关职责需要移动时，只移动该垂直切片；无关代码保持原位。

## 执行：确认后做最小安全迁移

获得明确确认后，按以下顺序实施：

1. 先确认现有行为测试和关键调用方测试是否已覆盖要迁移或简化的行为；覆盖不足时补测试并先观察测试能够暴露问题，覆盖充分时记录并运行已有测试。
2. 先复用原生能力或已有模块；确需新逻辑时，只实现当前真实调用需要的最小 API。
3. 提取一个职责单元，保留必要的兼容导出；不要同时重命名、格式化或搬迁无关模块。
4. 每次小迁移后运行相关测试、类型检查和调用方验证。
5. 最后复查 diff：是否新增了第二层 wrapper、默认工厂、空接口、永远不会切换的策略，或新的大杂烩入口。

### 上线优先的例外

若拆分会显著扩大当次风险，可以在原文件内完成经过测试的最小改动，但必须：

- 不复制已有规则或重新实现已有平台能力；
- 把新逻辑收束为局部、可命名的单元，避免继续扩散耦合；
- 在交付中标明“为何本次不拆”和下一次出现什么触发条件时应治理。

## 常见误区

| 借口 | 处理原则 |
| --- | --- |
| “以后很多地方可能会用” | 等第二个真实调用场景，且确认它的变化方式不同。 |
| “先加一个通用 utils” | 先问它是否跨领域且无业务语义；否则放回领域模块。 |
| “文件太大，全部拆掉” | 先移动本次相关职责，避免无关迁移带来的回归。 |
| “新依赖看起来更专业” | 先说明原生能力和既有依赖为何不够，再评估新增成本。 |
| “赶时间，先堆进去” | 走最小变更分支，但不复制逻辑、不新造框架，并保留验证。 |

## 交付前自检

- 是否先查过标准库、平台能力、现有依赖和现有模块？
- 新抽象是否已有至少两个真实且变化不同的调用者？
- 若使用单调用者抽象例外，是否已说明框架契约、对外 API、基础设施边界或测试/安全需求，以及对应成本？
- 删除前是否已核实静态调用、运行时注册/配置、导出面、部署入口和兼容承诺？
- 拆分是否按职责，而非按任意行数或文件大小？
- 是否没有夹带无关整理、重命名或架构升级？
- 未确认时是否只停在审计/方案？已确认后是否有行为测试和调用方验证？
- 最终 diff 是否比原方案更短、更直接，且没有新增大杂烩入口？
