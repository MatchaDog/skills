# Frontend

## 既有项目

沿用现有构建工具、包管理器、样式系统、数据层、状态库、目录和锁文件。局部任务不得引入平行方案，也不得要求用户重新确认仓库已经表达的选择。

## 新项目默认

只有用户、框架和组织都没有既定选择时，使用以下团队默认：

- TypeScript 严格模式。
- Rsbuild、Biome、pnpm、Tailwind CSS。
- 组件库优先复用成熟方案，不重复造轮子。

默认是起点，不是迁移目标。依赖选择前仍需确认框架兼容性和真实需求。

## 类型

- 新代码不使用隐式或逃避检查的 `any`；优先 `unknown`、泛型、判别联合或项目已有 Schema 推导。
- 不为当前任务清理无关的历史 `any`。
- API、表单和状态边界使用明确类型；生成类型不手工复制或修改。

## 数据访问与状态

- 优先复用生成客户端、框架原生能力或现有请求层。
- 只有共享鉴权、错误转换、重试等需求真实出现时才增加包装层；不为单个调用预建全局请求基础设施。
- 远端状态使用项目已有方案，本地 UI 状态优先保持局部；没有跨组件需求时不引入全局 Store。
- 不默认安装 axios、ky、TanStack Query、Zustand、Jotai 或其他数据与状态依赖。

## 结构与界面

- 遵循框架和项目现有结构；新项目从承载当前功能的最少目录开始。
- 代码按业务能力就近放置；出现第二个真实消费者或明确独立职责时再提取共享模块。
- 使用语义化 HTML 和原生交互元素，保持基本可访问性。
- 沿用项目样式系统，不在同一项目引入第二套 CSS 方案。

## 配置与验证

- Node 版本、包管理器和脚本沿用现有声明；新项目选择一种版本声明方式即可，不强制同时创建 `.nvmrc`、`engines` 等重复配置。
- `.npmrc`、镜像和私有 Registry 只在用户或组织明确需要时配置。
- 至少运行项目已有测试、TypeScript 检查、Lint 和构建。
