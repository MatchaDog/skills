# Universal

## 最高优先级

1. 用户明确选择。
2. 现有项目、框架和组织约定。
3. 本 Skill 的新项目默认。

既有项目不得仅为“统一规范”迁移工具、重排目录、重命名文件、补历史注释或增加根配置。偏离默认不需要专门写决策文档，除非它影响长期维护。

## 测试驱动开发

**REQUIRED SUB-SKILL:** 使用 `superpowers:test-driven-development`。

该子 Skill 不可用时停止实现，按主 `SKILL.md` 只安装它，不以手写简化版替代。

功能、Bug 修复、重构和行为变化必须执行：

1. **RED**：先写最小行为测试并运行。语法、依赖、环境错误或一开始就通过都不算 RED。
2. **GREEN**：只写使测试通过的最小实现，再运行相关回归。
3. **REFACTOR**：全部通过后才整理结构，并持续保持绿灯。

RED 证据至少报告测试命令、退出状态和关键断言。没有测试体系时，只建立当前任务需要的最小入口；赶时间、改动小和手工验证不是例外。

不得擅自删除、回退、覆盖、移动、stash 或清理用户改动。已有半成品不能作为反向补测试的参考；优先在不改变当前工作区的基线上建立 RED，任何工作区状态变更都先取得确认。

一次性原型、生成产物、纯文档或纯配置变更只有经用户明确确认才可跳过；功能、Bug、重构、行为变化、赶时间和缺少测试体系都不是例外。

## 命名与注释

- 沿用现有项目和框架命名；没有约定时，新建普通业务文件默认使用 kebab-case。
- 不为统一命名重命名现有文件。
- 注释只记录类型和代码无法表达的业务约束、前置条件、副作用、兼容原因与临时方案。
- 临时方案使用 `TODO`、`FIXME` 或 `HACK`，写明原因和退出条件。

## 文档与需求

- README 只需让维护者能理解目标、启动、配置和主要验证命令。
- 业务语义、权限、状态、公共契约或跨实现一致性变化时，更新当前有效来源和验收规则。
- 高风险且需要长期保留的技术决策才建立设计或决策文档。
- 不为一次变更同时创建需求说明、设计、计划、追踪矩阵和变更记录。
- 第三方知识库默认只做版本化事实来源的单向投影。

中文文本确需清理 AI 腔时按需安装：

```bash
npx skills add https://github.com/MatchaDog/stop-slop-cn --skill stop-slop
```

`stop-slop` 不得改变技术事实、需求语义、字段、错误码或验收规则。

## 交付门禁

- 当前目标可独立验收，Diff 没有无关迁移、重构或格式化。
- TDD 具备有效 RED 证据，GREEN 和相关回归通过。
- 存在系统边界时，契约来源与实现一致，生成产物未被手工分叉。
- 运行项目已有测试、类型检查、Lint 和构建，并报告实际结果。
- 高风险变更已经确认影响、兼容性和恢复路径。
