# Universal

所有项目都必须执行本文件。这里定义跨技术栈的基础约束，以及与项目风险相称的文档和需求追踪规则。

## Skill 协作

只按任务需要安装其他 Skill。开发新功能、Bug 修复、重构或行为变化时，`superpowers:test-driven-development` 是必要的开发纪律，不代表启用 Superpowers 的其他流程。

编写、编辑或审阅中文文档，确实需要清理 AI 腔、套话、公式化结构或翻译腔时，按需安装 `stop-slop`：

```bash
npx skills add https://github.com/MatchaDog/stop-slop-cn --skill stop-slop
```

`stop-slop` 不得改变技术事实、需求语义、API 字段、错误码或验收规则。使用已有成熟 skill 时，只写明安装命令和协作边界，不重复编写对方已经维护的详细内容。

## 命名规范

- 文件和文件夹命名必须使用 kebab-case。
- 不允许出现大写字母、PascalCase、camelCase、snake_case。
- 示例：使用 `user-profile.tsx`，不要使用 `UserProfile.tsx`、`userProfile.tsx`、`user_profile.tsx`。
- 框架约定冲突时，优先寻找框架支持的 kebab-case 写法；确实无法避免时，在文档中记录原因。

## 注释规范

- 注释默认使用中文编写，便于团队知识传承、Code Review 和企业内部知识库检索。
- TypeScript、JavaScript、Vue、React、Node.js 项目优先使用 JSDoc/TSDoc 风格注释。
- 公共函数、公共类型、导出的 hooks、store、API client、service 方法、复杂组件 props 必须补充 JSDoc/TSDoc。
- 注释优先解释“为什么这样做”“业务规则是什么”“边界条件是什么”，不要重复代码已经能表达的“做了什么”。
- 复杂业务判断、兼容逻辑、权限规则、错误码映射、缓存策略、重试策略需要写中文注释。
- 禁止为了满足形式写逐行注释；简单变量赋值、显而易见的 JSX/模板结构不需要注释。
- 临时方案必须用 `TODO`、`FIXME` 或 `HACK` 标记，并写清楚原因、后续处理方式和关联需求或任务。

JSDoc/TSDoc 示例：

```ts
/**
 * 根据当前用户权限过滤可见菜单。
 *
 * 业务约束：
 * - 超级管理员可以看到全部菜单。
 * - 普通用户只能看到已授权且未隐藏的菜单。
 *
 * @param menus 原始菜单树
 * @param permissions 当前用户权限码集合
 * @returns 过滤后的菜单树
 */
export function filter-visible-menus(
  menus: MenuItem[],
  permissions: PermissionCode[],
): MenuItem[] {
  // ...
}
```

## 测试驱动开发

**REQUIRED SUB-SKILL:** 使用 `superpowers:test-driven-development`。

功能、Bug 修复、重构和行为变化必须遵循 RED–GREEN–REFACTOR：

1. **RED**：先写一个描述目标行为的最小测试，实际运行并确认它因行为尚未实现而失败。语法、依赖、环境或配置错误不算有效 RED；测试一开始就通过也不算。
2. **GREEN**：只写让该测试通过的最小实现，运行目标测试和相关回归测试。
3. **REFACTOR**：全部测试通过后才整理结构，并在重构过程中保持测试通过。

Bug 必须先有自动化复现测试。项目没有测试体系时，先建立当前任务所需的最小测试入口；赶时间、改动小、手工复现或历史上没有测试都不是例外。

开始 TDD 前已经存在的半成品实现不能作为实现参考。不得擅自删除、回退、覆盖、移动、stash 或清理用户已有及未提交改动；优先在新 worktree、临时副本或其他不改变当前工作区状态的基线上建立 RED。任何会改变当前工作区状态的隔离操作都必须先取得用户确认。一次性原型、生成代码和纯配置变更只有经用户明确同意才可跳过。

RED 证据至少报告测试命令、退出状态和关键断言失败摘要；只口头声称“已经失败”不能进入 GREEN。

## 文档分级

文档用于保存代码和契约无法表达的长期事实，不按固定目录批量创建空模板。

- **局部实现变化**：样式、普通文案、重构、依赖升级、生成代码刷新。通常只修改代码和测试；必要时更新 README 或发布记录。
- **业务语义变化**：权限、状态流转、API 语义、数据解释、跨端能力。更新当前有效需求来源和验收规则；需要跨文档检索时加载 `maintaining-requirement-traceability`，维护最小需求索引。
- **高风险技术决策**：架构、数据迁移、事务边界、安全或不可逆变更。先给出方案并等待确认；只有决策需要长期保留时才建立设计或决策文档。

已有文档优先复用。不要为一次变更同时创建需求说明、设计文档、任务拆分、追踪矩阵和变更记录来重复同一信息。

## 需求、契约与代码

- 项目当前有效的版本化来源保存业务语义；接口、事件、Schema、命令或其他系统边界保存相应契约。
- 需求绑定稳定标识和必要锚点，具体实现通过项目已有的代码图、LSP、静态分析或检索工具动态发现，不手工维护完整文件路径。
- 不同客户端、服务、语言或运行时实现同一能力时，共享业务语义和验收规则，但允许实现结构不同。
- 第三方知识库默认只作为 Git 内容的只读投影，不参与双向编辑。

## 最小项目文档

根目录 `README.md` 应按实际情况说明项目目标、本地启动、环境变量、常用脚本和必要入口。接口、部署、用户指南或决策记录只在项目确实需要时创建，可合并到现有文档。

文档默认使用中文，写清适用范围和当前状态。需求或契约变化时更新其事实来源；实现细节能从代码和工具可靠推导时，不重复抄写。

## 项目交付门禁

交付前检查：

- 目录和文件名符合 kebab-case，框架强制约定的例外已有说明。
- 关键公共 API、复杂业务逻辑和临时方案有必要的中文 JSDoc/TSDoc 或中文注释。
- README 足以启动和维护项目，没有无内容的占位文档。
- 业务语义变更能追溯到当前需求来源和验收规则。
- 存在系统边界契约时，使用项目选定的 OpenAPI、GraphQL Schema、AsyncAPI、消息 Schema 或其他契约来源约束；生成产物不得被手工分叉。
- 功能、修复、重构和行为变化具备包含命令、退出状态和关键断言摘要的 RED 证据，并完成 GREEN 与回归验证。
- 高风险变更已经过方案确认，并覆盖测试、兼容性、回滚或恢复路径。
