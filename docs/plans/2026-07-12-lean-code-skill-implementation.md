# 精简代码 Skill 实施计划

> **面向 Codex：** 必须使用 `superpowers:executing-plans`，逐项执行本计划。

**目标：** 创建一个通用 Skill，使 Codex 在处理自造轮子、过度抽象和巨型模块问题时先审计、后方案、确认后执行。

**架构：** Skill 保持自包含，仅包含触发条件、审计分类、方案模板、执行门槛和验证清单。它将现有 `ponytail` 的减法审查能力作为可选协作项，不依赖该插件可用。

**技术栈：** Markdown、Codex Skill frontmatter。

---

### 任务 1：建立 Skill 与触发描述

**文件：**

- 新建：`skills/lean-code/SKILL.md`

**步骤 1：写入已验证的触发情境**

覆盖“自造轮子”“过度设计”“巨型文件”“大杂烩 utils”“拆分模块”“减少依赖/抽象”等表达；不把普通功能开发误触发为治理任务。

**步骤 2：写入最小工作流**

以“审计 → 方案 → 确认 → 执行 → 验证”为主线；上线紧急需求走最小变更分支。

**步骤 3：写入安全例外和删除门槛**

将“两个真实调用场景”定义为默认抽象门槛；对于框架契约、稳定对外 API、基础设施边界和可验证测试/安全需求，要求在方案中记录例外理由与成本。删除前核实运行时注册、配置、导出、部署和兼容面。

**步骤 4：校验元数据**

运行：`sed -n '1,80p' skills/lean-code/SKILL.md`

预期：仅含 `name`、`description` 两个 frontmatter 字段，且描述以 `Use when...` 开头。

### 任务 2：场景回测

**文件：**

- 修改：`skills/lean-code/SKILL.md`（仅在回测发现缺口时）

**步骤 1：使用三类基线请求回测**

覆盖巨型服务赶工、单调用点重试框架和大杂烩工具文件。

**步骤 2：检查关键行为**

确认回答会先检查复用能力、按职责提出有限拆分、列出兼容与测试方案，并在用户未确认时不实施。

**步骤 3：若发现逃逸路径则补充约束**

只补足回测暴露的缺口，避免把 Skill 写成泛泛的“最佳实践”清单。
