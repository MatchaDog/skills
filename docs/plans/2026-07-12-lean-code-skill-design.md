# 精简代码与模块边界 Skill 设计

**目标：** 将“减少自造轮子、避免过度抽象、治理巨型文件”的通用工作流固化为一个可复用 Skill。

## 背景与基线

无 Skill 基线使用了三类高压请求：在 1,800 行服务中赶工追加功能、为一个调用点设计完整重试框架、向 900 行 `utils` 大杂烩追加领域规则。现有仓库约束已能促使代理先出方案再执行；但它没有专门、可检索地要求代理检查标准库/既有依赖、拒绝单一场景的投机抽象，也没有明确领域模块与技术工具的边界。

## 选择

采用一个自包含的 `skills/lean-code/SKILL.md`，而非引入新的插件或第三方依赖。

- 默认只读审计，避免把“发现复杂度”误变成未经授权的全仓重构。
- 审计结论固定分为删除、复用、延后抽象、按职责拆分、保持现状五类。
- 输出方案后明确等待用户确认；确认后才允许改代码。
- 对线上紧急修复保留最小变更分支：可以暂时不拆文件，但仍要检查已有能力、限定改动范围、留下后续治理项。

## 安全边界

- 删除结论必须同时核实静态调用、运行时注册或配置、导出面、部署入口与兼容承诺；任何一项证据不足时不删除。
- “两个真实且变化不同的调用场景”是默认抽象门槛，而非机械禁令。框架契约、稳定对外 API、基础设施隔离，或可验证的测试/安全需求可构成例外，但需要在方案中说明成本和必要性。
- 测试策略先复用既有行为与调用方覆盖；覆盖不足时才新增测试，避免对纯移动或纯删除夹带无关改动。

## 不做的事

- 不以单一行数阈值强制拆文件；职责混杂和变更耦合才是主判断依据。
- 不要求为了“干净”迁移无关代码。
- 不把第二个尚未出现的调用场景当作抽象理由。
- 不替代正确性、安全性或性能专项审查。

## 验收

1. 面对单一调用点的策略/工厂框架需求，Skill 要求先检查原生能力、现有依赖与真实变化点。
2. 面对巨型 Service 或 `utils`，Skill 要求按职责和领域划分目标边界，且不默认全量搬迁。
3. 未获得明确确认时，Skill 只输出审计或方案，不修改代码。
4. 获得确认后，Skill 要求以既有或新增的行为测试、最小迁移和兼容验证为门槛。
5. 删除建议在动态入口、公开导出或兼容面未经核实前必须保持为待验证项。

## 回测结果

已使用同一组场景读取 `skills/lean-code/SKILL.md` 后复测：

- 1,800 行 Service 的赶工请求被识别为上线优先的方案模式；结论是只审计和限定局部规则切片，不立即改代码或发起大拆分。
- 单一 Node 20 `fetch` 调用的重试需求先检查原生能力、既有依赖、幂等性与真实调用数量；结论是延后策略/工厂框架。
- 900 行 `utils` 的领域规则追加请求被识别为方案模式；结论是活动资格和订单折扣分别进入其领域模块，不继续扩张工具大杂烩。
